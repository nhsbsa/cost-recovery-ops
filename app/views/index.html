Version 28 / S1 (August - September 2024)

S1 (MVP - Version 2) In this version you will see a pivot in our approach from a GDS citizen-style service to an admin-style portal whereby the user must first search for a person, and if no results are found, then a new record must be created. Once the record has been created, the user can then update personal details, add entitlements and add dependants. Other processes such as enquries and cancellations are yet to be worked on.

For site flows, wireframes, and general design thinking for S1 (MVP Version 2) visit the Content and Design Miro board.

What we created:

S1 - Search for a person, view their personal record or create one

  • Search results - Person record found
  • Search results - no results found

What we plan to test:

  • All of the above journeys

What we found:

  • Cost Recovery Officers appreciate our approach to the S1 Registerations and Cancellations work and like the direction we're heading in. They believe that the stages of searching for a person, creating a new person record, and being able to view and update personal details is simple and easy to use.
  • At this stage, there isn't a user need for the Person Overview screen, rather it is a nice-to-have feature which would simply enhance the portal. Therefore, we plan to drop this screen from the MVP.
  • Cost Recovery Officers also like the Entitlements and enquiries section but think it needs improvement, particularly around the type of information that is displayed as part of the overview. As part of improvement work, they would like to be able to clearly see the status of each entitlement e.g., registered or cancelled.
  • Cost Recovery Officers believe that Case History should contain system updates e.g., record created, first name updated, email added, etc. whereas Notes should be used by Cost Recovery Officers to provide context, and effectively 'paint the bigger picture' of the person's record. For example, notes could include anything from a complaint, to information regarding a conversation that might have taken place between one CR Officer and the individual. It may also contain details around any letters or documents that have been sent or received.
  • Individual Notes tabs should be removed from both the Personal details section and individual entitlements. Instead, a separate Notes tab should be added to the left-hand navigation menu so that the notes are against the entire person record, as opposed to individual entitlements. This will help CR and Claims officers understand the wider context of each person's record and attached details.

Any known issues:

  • Cost Recovery Officers will very rarely input the UK NINO number when creating the person record so as part of the next version of the prototype, this data point should be removed from the Personal details data entry screen and instead, left for CR Officer to update in the Personal details section of the person record, if and when they receive this information. For example, from a third party referral, on the back of receiving the FAS1Q or DL1609.
